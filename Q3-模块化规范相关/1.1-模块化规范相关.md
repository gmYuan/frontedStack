## webpack内部原理

### 一 基础含义 与 模块化发展概述

Q1 什么是模块化 + 为什么前端会引入模块化

A: 

1 什么是模块化
  - 模块化是指把 一个复杂的系统 分解为 多个独立的模块
  - 每个模块都是独立的作用域，有自己的变量、函数等私有成员
  - 模块之间通过导入导出进行通信，从而形成 依赖关系


2 为什么前端需要模块化：

2.1 解决早期开发遇到的 以下问题：
  - 命名冲突 和 全局变量污染
  - 文件依赖不明确，文件加载顺序 需要手动管理
  - 代码 功能职责不单一，导致 难以维护
  

2.2 引入后的改善/带来的好处：
  - 作用域隔离，避免命名冲突
  - 显式声明了依赖关系，便于管理
  - 功能职责单一，便于维护
  - 拆分后，也便于实现 按需加载、并行加载 等优化策略


------------------------------------------------------
Q2 模块化的具体实现，大致经历了哪些阶段

A: 前端模块化经历了以下几个主要发展阶段：

1 早期阶段1- 全局函数
  - 原理：直接定义全局函数
  - 问题：命名冲突、依赖关系不明确
  
```js
function fn1() { /* ... */ }
function fn2() { /* ... */ }
```

2 早期阶段2- 命名空间模式（namespace）
  - 原理：将相关函数放入对象中，本质是对象级别的命名空间
  - 问题：无法实现 作用域隔离，在外部还是可以 随意修改内部状态
  
```js
var MyApp = {
  fn1() { /* ... */ },
  fn2() { /* ... */ }
};
```

3 早期阶段3- IIFE（立即执行函数）+ 闭包
  - 原理：通过闭包实现 私有变量， 通过传参 暴露依赖关系
  - 问题1：无法自动管理依赖关系，需要手动维护顺序
  - 问题2：无法实现异步加载，难以实现按需加载

```js
var Module = (function($) {
  var private = 'private';
  return {
    public: function() { /* ... */ }
  };
})(jQuery);
```


4 社区模块化规范- CommonJS
  - 特点：同步加载，适合服务器端
  - 是 Node.js 采用的模块规范

```js
const fs = require('fs');
module.exports = { /* ... */ };
```

5 社区模块化规范- AMD / CMD
  - 专注解决浏览器端的模块化
  - AMD（RequireJS）：依赖前置，提前执行
  - CMD（SeaJS）：依赖就近，延迟执行

```js
// AMD
define(['jquery'], function($) {
  return { /* ... */ };
});

// CMD
define(function(require, exports, module) {
  var $ = require('jquery');
  exports.foo = function() { /* ... */ };
});
```

6 官方模块化规范- ES Modules
  - 特点：静态分析，支持 Tree Shaking
  - 是 ECMAScript 官方模块化方案
  
```js
import { foo } from './module';
export const bar = /* ... */;
```

7 现代化构建工具（Webpack/Rollup/Vite 等）
  - 统一处理各种模块规范
  - 提供更多优化特性（代码分割、按需加载等）

```js
// webpack.config.js
module.exports = {
  entry: './src/index.js',
  output: { /* ... */ }
};
```


### 二 CommonJS模拟实现


Q1 CommonJS规范 的基础使用语法 是什么

A: 

1 通过 module.exports 导出模块

```js
// num.js
var a = 1
var b = 2
var add = function (){
  return a + b
}

// 导出方法1.1- 覆盖引用地址
module.exports = add;

// 导出方法1.2- 覆盖引用地址
module.exports = {
  add
};

// 导出方法2.1- 直接添加属性，不修改引用地址
module.exports.add = add
module.exports.name = 'num'

// 导出方法2.2 - 等价于 2.1
Object.assign(module.exports, {
  pow: Math.pow,
  sqrt: Math.sqrt
});

// 导出方法3- 使用 exports 快捷方式
// 在Node里，exports 就是 module.exports 的引用
exports.add = add


// 注意：以下是错误导出方法

// ❌ 会断开 exports 和 module.exports 引用关系
// exports = { add }

// ❌ 同上，不要直接给 exports 赋值  
// exports = add     
```


2 通过 require 导入模块：require() 返回的就是 引入模块的 module.exports 对象

```js
// 导入方法1- 直接导入
var num = require('./num')


// 导入方法2.1- 解构赋值导入
const { add, name } = require('./num')

// 导入方法2.2- 导入后重命名
const { add: sum, sqrt: squareRoot } = require('./num')


// 导入方法3- 动态导入
if (condition) {
  const mathModule = require('./num')
  mathModule.add()
}
```

Q1 你知道哪些模块化标准，各自有哪些特点

A: 主流的模块化标准包括以下几种：

1. CommonJS (CJS)
   - Node.js 默认模块规范
   - 同步加载，运行时加载
   - 使用 require 导入，module.exports 导出
   - 输出值的拷贝，不会受到模块内部后续变化的影响
   - 适用于服务器端开发

2. ES Modules (ESM)
   - ECMAScript 官方模块化规范
   - 静态分析，编译时加载
   - 使用 import/export 语法
   - 输出值的引用，支持 Tree Shaking
   - 浏览器原生支持，现代前端开发首选

3. AMD (Asynchronous Module Definition)
   - 异步模块定义规范
   - 专门用于浏览器端
   - 使用 define 定义模块，require 加载模块
   - 依赖前置，提前执行
   - 代表实现：RequireJS

4. CMD (Common Module Definition)
   - 通用模块定义规范
   - 专注于浏览器端
   - 使用 define 定义模块
   - 依赖就近，延迟执行
   - 代表实现：SeaJS

5. UMD (Universal Module Definition)
   - 通用模块定义规范
   - 兼容 AMD、CommonJS 和全局变量
   - 在浏览器和服务器端都可以运行
   - 常用于打包工具的输出格式
   - 适合作为库的发布格式

Q2.1: CJS 和 ESModule 的主要区别:

A: 

1. 语法差异：
  - CJS: 使用 `require()` 导入，`module.exports` 导出
  - ESM: 使用 `import` 导入，`export` 导出

2. 加载时机：
  - CJS: 运行时动态加载
  - ESM: 编译时静态分析，支持 Tree Shaking

3. 加载方式：
  - CJS: 同步加载
  - ESM: 异步加载

4. 导出值：
  - CJS: 导出值的拷贝
  - ESM: 导出值的引用（绑定）

5. 循环依赖处理：
  - CJS: 返回未完成的导出对象，可能出现未完成的导出
  - ESM: 通过模块依赖图和动态绑定处理

6. 缓存机制：
  - CJS: 缓存模块实例
  - ESM: 缓存模块命名空间对象

7. 使用场景：
  - CJS: 主要用于 Node.js 服务端
  - ESM: 浏览器原生支持，现代前端开发推荐使用
